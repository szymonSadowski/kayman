## What Went Well

- **End-to-end pipeline shipped and working in beta.** Transcribe → summarize → export → notify is composable, with each stage following a clean `runStage(input) → output | throws PipelineError` interface.
- **Detached spawn pattern works.** `kayman stop` returns within 1s while runner continues in background — meets NFR3.
- **Notion retry logic is robust.** 429/5xx exponential backoff (base 1s, cap 8s, max 3 attempts), 401 fails fast with actionable message. Tested.
- **Personal Spotlight kept in shared package.** `applySpotlight` reused from `@kayman/shared` rather than re-implemented.
- **Test discipline held under expansion.** Stories 3.6–3.9 added 3 new commands + 480 lines of test code in a single commit. No coverage backslide.
- **Failure preservation is real.** `transcript.txt` survives any pipeline failure; `summary.json` exists before export attempts; `.exported` marker enables `kayman retry` flow.
- **`kayman verify` rounded out the user setup story.** Health check covers config, whisper binary/model, AI creds, Notion access — no short-circuit, all checks always run.

## Challenges & Growth Areas

### 1. Post-merge fix waves on external SDK upgrades
After story 3.3 landed, multiple commits chased ai-sdk v6 compat:
- `998aeee` — `LanguageModel` type + `Output.object` for structured output
- `4915617` — `maxOutputTokens` rename
- `217814b` — runner path + summarization/export improvements

**Root cause:** ai-sdk v6 was a breaking release; the planning phase didn't pin a version or note the v5→v6 migration surface.

### 2. TypeScript module resolution surprise
`35d1f8e` had to add `.js` extensions to every relative import across CLI source/tests after switching to `Node16` moduleResolution (`3123b5a`). Touched 14 files.

**Root cause:** moduleResolution change made implicitly. The follow-up cleanup commit was mechanical but distracting.

### 3. Story file hygiene drift
- Stories 3.1–3.5 have story files; 3.6–3.9 do not.
- Story 3.1 was created **retroactively** to document existing code.
- Stories 3.2–3.5 still show `Status: review` in their files even though they're merged to main.
- No `sprint-status.yaml` exists, so the retrospective workflow can't auto-detect epic state.

**Root cause:** Sprint hygiene wasn't part of the story-cycle muscle memory. Story files became optional rather than load-bearing.

### 4. Sprint board is stale
`_bmad-output/sprint-board.md` still lists 3.6–3.9 under **Todo** even though they're merged.

### 5. Mid-epic scope expansion
Stories 3.6–3.9 were added to Epic 3 mid-flight (commit `207d2e3`), bundled with brand-new Epic 5 and Epic 6. Epic 3 originally ended at 3.5 (the runner). The expansion was healthy in outcome but blurred "epic done" criteria.

### 6. Runner path / config bugs only caught at integration
- `217814b` fixed runner path issue
- `21542eb` fixed missing Project relation when summary.project matches config
These weren't caught by the unit test suite — they needed real beta usage to surface.

## Key Insights

1. **Unit tests passed but integration bit us.** The summarize/export tests mocked the AI SDK and Notion client cleanly, but real ai-sdk v6 + real Notion DB schema were the actual contracts. Need at least one smoke test against a real provider before shipping.

2. **External SDK pinning matters.** When story 3.3 was implemented, the ai-sdk version was floating. v6 broke us. Lesson: pin SDK majors in `package.json` and explicitly bump.

3. **Node16 moduleResolution should have been a planned migration.** Fixing it as a one-line config change generated 14-file follow-up.

4. **Story files only pay off if maintained.** Half-maintained story files are worse than none — they create false confidence about what's documented.

5. **The "expand the epic" reflex was correct here.** 3.6–3.9 (tags, list, retry, verify) were the things that made the pipeline actually usable end-to-end. Adding them inside Epic 3 instead of carving out Epic 3.5 was the right call — they unblocked Epic 4's Raycast work.
